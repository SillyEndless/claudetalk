# 爆肝48小时，在IM群聊中的Multi-Agent最佳实践

最近花了两天时间，把多 Agent 协作这件事在 IM 群聊里跑通了。

踩了不少坑，也想清楚了一些事情。这篇文章不讲某个具体工具，讲的是在 IM 群聊里做多 Agent 协作这件事本身——为什么要这么做，怎么设计，以及哪些地方会出问题。

## 为什么单个 AI 搞不定复杂任务

用 AI 写代码，最头疼的不是它不够聪明，而是不可控。

你让它做个待办事项应用，它可能从需求分析写到前端框架，再写到数据库设计，最后还顺手给你写一堆测试用例。听起来挺厉害，但实际上经常出问题——要么上下文太长忘了前面的要求，要么各个部分之间逻辑不一致，要么就是想做的事情太多，最后什么都没做好。

这就像让一个人同时当产品经理、架构师、开发、测试，还要做项目管理。现实中没人这么干，因为根本做不好。

那为什么 AI 就要这么做呢？

其实不用。我们可以把开发团队的角色分工引入到 AI 协作里，让多个专业化的 AI 各司其职。就像真正的开发团队一样，有人负责需求，有人负责设计，有人负责写代码，有人负责测试。

这就是 Multi-Agent 的核心思路：**用分工代替全能，用专注代替万能**。

---

## 为什么选 IM 群聊作为协作场所

多 Agent 协作的实现方式有很多，为什么偏偏选 IM 群聊？

答案是：**透明度和可干预性**。

想象一下，你在群里跟团队沟通。你说"我想做个应用"，然后 @项目经理。项目经理拆解任务，@产品经理去定义标准，产品经理 @架构师做设计，架构师 @开发写代码，开发写完了 @测试去验证。

整个过程你都能看到，随时可以插话。觉得需求不对，直接说；觉得设计有问题，马上提出来；看到代码有 bug，立刻指出来。

这种体验在传统的 AI 对话里很难实现。你跟一个 AI 说"做个应用"，它就开始一顿输出，等它说完了你才发现方向偏了。那时候再改，要么重新开始，要么就是各种补丁。

IM 群聊的价值在于：**它把多 Agent 协作的底层逻辑，用最直观、最自然的方式呈现出来。既自动化，又可控。**

## 机器人之间怎么互相@协作

在 IM 群聊里实现多 Agent 协作，最关键的是让机器人之间能够自动流转。理想的状态是：PM 机器人说完，直接 @ 前端开发机器人，前端开发机器人自己接过去处理。你在旁边看着，觉得哪里不对随时插一句。

### 本项目的实现机制

IM 平台（如飞书、钉钉）有个限制：**机器人发的消息，其他机器人收不到**。这是平台层面的设计，防止机器人之间无限循环触发。

所以没法靠 IM 平台的事件推送来实现机器人间通信。本项目的解决方案是用**共享文件做消息队列**：

1. **PM 机器人发消息**：PM 处理完需求，回复里 @ 了前端开发机器人。系统检测到 @ 标签，把这条消息写入 `bot_前端开发.json` 文件。

2. **前端开发机器人接单**：前端开发机器人定时扫描自己的消息文件，发现有新消息就处理，完成后从文件里删除。

3. **IM 群展示**：整个过程在群里看起来就是：PM 机器人说完，前端开发机器人的消息上出现一个 👌，然后开始回复。

**实际效果**：
整个协作过程完全透明，你可以看到每个机器人的发言，随时可以插话干预。

## 三个必须解决的工程问题

把这套东西跑通，有三个绕不开的工程问题。

### 1. 角色抢跑

比如开发想直接开始写代码，但架构师还没出设计。这时候应该阻止，而不是让开发自己发挥。

解决方案是给每个角色设置**前置依赖**：

```
开发工程师：必须等架构师的设计方案才能开始编码
测试工程师：必须等开发完成代码才能开始测试
```

这个约束写在角色的配置里，不是靠 prompt 约束，而是在流程层面就拦住。同时，项目经理会定期读取聊天消息，监控整个协作过程，确保每个角色按照正确的顺序执行。

### 2. 死循环

测试发现 bug，@开发修复；开发修复完，@测试复测；测试发现还有 bug，@开发再修复……如果这个循环一直转，就没完没了了。

本项目的解决方案是：**项目经理定期读取聊天消息，判断是否需要中断协作并请求人工干预**。

项目经理会监控整个协作过程，如果发现某个角色之间来回超过一定次数（比如 10 轮），就会主动介入：

```
项目经理：@用户 当前功能遇到技术阻塞，测试和开发之间已经来回修复 10 次，是否需要人工介入或降低需求标准？
```

这个机制很重要。Agent 不是万能的，遇到真正复杂的问题，还是需要人来拍板。

### 3. 上下文丢失

多 Agent 协作最大的隐患是上下文太长。每个角色都有自己的会话，但它们需要共享一些关键信息：需求文档、技术设计、里程碑状态。

如果这些信息只存在于某个角色的会话里，其他角色就看不到，协作就会出问题。

解决方案是**把关键信息持久化到文件**：

```
需求文档    → requirements.md
技术设计    → architecture.md
里程碑状态  → milestones.json
测试报告    → test_report.md
```

每个角色在完成自己的工作后，把关键输出写到对应的文件里。下一个角色接手时，先读文件，而不是依赖上下文传递。

这样即使某个角色的会话被清掉，整个协作也不会断。

## 权限隔离：不能靠 Prompt 约束

这是一个容易被忽视的细节。

很多人的做法是在 prompt 里写"你是产品经理，不要改代码"。但这不可靠——AI 有时候会"忘记"这个约束，或者在某些情况下绕过它。

更可靠的做法是**在工具调用层面做权限控制**：

```
产品经理：只能读取文件，不能写代码目录
架构师：可以写设计文档，不能写代码
开发：可以写代码，不能改需求文档
测试：可以写测试，不能改源码
```

这个权限是真实生效的，不是靠 prompt 约束。产品经理想改代码，工具层面就拦住了，Claude 想越界也做不到。

这个边界在实际使用中非常重要。你不希望 PM 角色在整理需求的时候顺手把你的代码改了。

## 模型分配：按需分配算力

不同角色对模型能力的要求不一样。项目经理主要做任务拆解，用轻量模型就够了；开发工程师需要写复杂代码，得用能力强的模型。

按角色分配模型，可以在保证质量的前提下控制成本。

## 人工干预的几种方式

这套机制里，人始终是在场的，不是完全甩手。

**直接插话**：机器人在群里讨论的时候，你随时可以 @ 某个机器人说"等等，这个方向不对"。本项目的实现机制是：机器人执行时会获取最新的历史消息记录（包括不是 @ 它的消息），同时在角色配置里要求把用户的发言作为最高优先级。这样被 @ 的机器人看到你的最新发言，就会按照你的最新指令来调整。

**接管某个角色**：如果觉得某个机器人的判断有问题，可以直接在群里说出你的判断，让另一个机器人基于你的判断继续往下走。

**终止协作**：发 `/new` 给某个机器人，清掉它的会话记忆，重新开始。

**观察整个过程**：所有机器人的发言都在群消息里，你可以完整看到它们是怎么协作的，哪个环节出了问题一眼就能看出来。

这才是 IM 群聊模式的核心价值——**既自动化，又可干预**。

## 踩过的坑

### 需求拆得太大

一开始我让项目经理直接处理类似"做一个完整的电商系统"，结果它拆出来的里程碑每个都很大，后面每个角色的工作量都超出了单次会话能处理的范围。

教训：**需求拆得越小越好**。每个里程碑最好是一个能在一次会话里完成的功能单元。宁可多几个里程碑，也不要让单个里程碑太重。

### 验收标准不够具体

产品经理定义的验收标准如果太模糊，比如"用户体验要好"，测试工程师就不知道怎么验，开发也不知道做到什么程度算完成。

教训：**验收标准必须可测试**。每一条都要能回答"怎么验证这条是否满足"。

### 上下文没有及时保存

有一次协作到一半，某个角色的会话因为太长被截断了，结果它"忘记"了之前的需求，开始按自己的理解往下走，最后产出的东西和需求完全对不上。

教训：**关键信息要及时写文件**。不要依赖会话上下文来传递重要信息，文件比会话可靠得多。

### 机器人之间的循环调用

多个机器人之间可能会出现循环调用的情况，导致协作无法推进。

这个问题在前面"死循环"部分已经详细说明了：项目经理会定期读取聊天消息，监控整个协作过程。如果发现协作陷入循环（比如某个角色之间来回超过一定次数），项目经理会主动请求人工介入。

这里的教训是：**不要试图靠 prompt 约束来防止循环**，而是通过项目经理的主动监控来发现和解决问题。

## 最佳实践示例：里程碑式协作

前面讲了这么多理论和工程问题，最后用一个完整的例子来说明多 Agent 在 IM 群聊里是怎么协作的。

### 角色设计

把开发团队的分工搬进来，五个角色基本够用：

- **项目经理（PM）**：拆解需求，分配任务，不写代码，只负责协调
- **产品经理（PD）**：定义清晰的验收标准，确保需求可测试、可验证
- **系统架构师**：做技术选型和架构设计，不写代码，只提供设计方案
- **开发工程师**：根据设计方案编写代码，保证代码质量和可维护性
- **测试工程师（QA）**：根据验收标准进行功能测试，报告 Bug 并验证修复

### 里程碑式执行流程

做一个待办事项应用，整个流程在 IM 群聊里是这样的：

```
你：@项目经理 我想做一个待办事项应用

项目经理：收到。为了快速迭代，我们把项目拆成几个里程碑：
  里程碑 1：基础增删查（本地存储）
  里程碑 2：数据库持久化
  里程碑 3：用户登录

现在开始里程碑 1。
@产品经理 请定义"里程碑 1"的验收标准

产品经理：验收标准如下：
  - 用户能看到输入框和添加按钮
  - 点击添加后列表里出现新条目
  - 每个条目后面有删除按钮
  - 刷新页面数据不丢失

@系统架构师 请做技术设计

系统架构师：技术栈 React + Vite + TailwindCSS，状态管理 useState，封装 useLocalStorage Hook。
@开发工程师 请写代码

开发工程师：代码已生成，构建成功。
@测试工程师 请测试

测试工程师：✅ 测试通过。
@产品经理 请验收

产品经理：✅ 验收通过。
@项目经理 里程碑 1 已完成

项目经理：🚩 里程碑 1 已完成。
[ 🚀 进入里程碑 2（数据库） ] [ 🛑 暂停/调整 ]
```

这个例子的关键点在于：

1. **里程碑式拆解**：把大需求拆成小里程碑，每个里程碑都能在一次会话里完成
2. **清晰的验收标准**：每个功能都有可测试的验收标准，避免扯皮
3. **自动化流转**：每个角色完成自己的工作后，自动 @ 下一个角色
4. **随时可干预**：你觉得哪个环节有问题，直接在群里说，机器人会根据你的反馈调整

这就是 IM 群聊里多 Agent 协作的最佳实践——**既自动化，又可控**。

## 实践建议

总结一下这两天跑下来的经验：

1. **小步迭代**：需求拆得越小越好，不要想着一次做完
2. **明确验收**：每个功能都要有清晰的、可测试的验收标准
3. **文档沉淀**：重要的决策和设计要保存到文件，不要只存在会话里
4. **权限控制**：每个角色只能操作自己负责的文件，不要靠 prompt 约束
5. **模型分配**：按角色的实际需求分配模型，控制成本
6. **人工监督**：关键节点要有人确认，不要完全甩手给 Agent
7. **防止死循环**：设置最大轮次，遇到阻塞及时介入

## 总结

IM 群聊 + 多角色协作，本质上是在模仿真实的开发团队。

每个角色各司其职，流程清晰规范。但最关键的，是整个过程是透明的，你可以随时监督，随时干预。

这更像是一个"有人在场的半自动协作"，而不是完全自主的 Agent 系统。机器人负责执行，人负责把控方向。这个定位在实际工作中反而更实用——你不需要完全信任 Agent，但可以让它帮你省掉大量重复的传话和格式化工作。

这才是 AI 辅助开发的正确打开方式——**既自动化，又可控**。
